这是我接触分布式系统领域读的第一篇论文，在论文的 Abstract 里面就说了 MapReduce 是一种编程模型，并且这种编程模型用来处理并生成大量的数据集。下面均为我个人对该论文的理解所写，如果有错，还希望指出。

用户定义一个 map 函数用来处理 key/value pairs, 并且在处理结束后生成一个中间的 key/value pairs. 这很好理解，举例说明，假设我有如下的数据集。

sup sup
duck duck duck thanks a bunch
thanks a bunch

当把此编程模型的 map 函数用在该数据集后，会产生如下的输出结果, 可以从下面的数据集看出来，括号内的第一个单词为 key , 第二个字母为该单词出现的次数。map 并不会对这些结果最汇总，只是单纯的对输入集做一个简单记录而已。每读到一个单词，就会标记出现了一次。论文里把下面的数据集叫做 intermediate key/value pairs。接下来，MapReduce 函数的编程模型就会把该数据集发送给 reduce 函数。

(sup, 1)
(sup, 1)
(duck, 1)
(duck, 1)
(duck, 1)
(thanks, 1)
(a, 1)
(bunch, 1)
(thanks, 1)
(a, 1)
(bunch, 1)

reduce 函数在接受到上述的数据集后，就会做一件事，论文里面是这样说的：accepts an intermediate key i and a set of values for that key. It merges together these values to form a possibly smaller set of values. 按照这样的理解，对于 WordCount 这样的业务需求，我在 reduce 里面所做的事情应该就是把相同 key 的 value 进行合并，也就是说求和。做一个极端思考: 假设此时，对于 key = sup 这样的数据在 A 机器上 [中国]，而对于 key = duck 这样的数据在 B 机器上 [澳大利亚]。那么在做最后结果汇总的时候就难免会因为网络传输所带来的序列化和反序列化的操作，那么这样会造成很大的 IO 开销以及带来各种问题，当然还有很多别的问题都会产生，比如时钟的问题，见这篇论文 Time, Clocks, and the Ordering of Events in a Distributed System，海底电缆被鲨鱼咬断等等等等各种问题，这也就是分布式系统领域困难的地方。

按照上面的说法，reduce 函数在处理之后，会产生如下的数据集, 这也是我们最后需要的结果。

(sup, 2)
(duck, 3)
(thanks, 2)
(a, 2)
(bunch, 2)

同样的，MapReduce 编程模型还可以用在很多别的场合，比如我需要统计网站不同网页的访问次数。

map 函数用来处理每次用来请求的地址，并且输出 (URL, 1)， 比如下面的：

(http://www.vip.com/, 1)
(http://www.vip.com/, 1)
(http://kid.vip.com/, 1)
(http://act.vip.com/22263.html, 1)
...
...

再经过 reduce 函数输出最后的统计结果

(http://www.vip.com/, 2)
(http://kid.vip.com/, 1)
(http://act.vip.com/22263.html, 1)
...
...

下面来看下 MapReduce 编程模型的执行图

从图中可以看出，用户会在本地写代码实现具体的业务逻辑，这个叫做 User Program. 然后打成一个 Jar 包发到远程的服务器去执行。在分布式系统中，通常有一个 Master 结点和多个 Worker 结点运行在集群上。用户提交作业到 Master 进程，并且在 Master 结点上启动程序。Master 进程会把 User Program 复制分发到做具体工作的各个 Worker 结点上。对应图中第一步 fork

按照论文里面的说法，当程序变成进程开始执行的时候，MapReduce 编程模型会把用户提供的原始数据集进行 自动的 分割，以形成 M 个数据集。也就是图中的 [split 0,1,2,3,4,…,M-1,M], 此时，Master 会选择空闲的 Worker ，并且把 Map Task 和 Reduce Task 分配到这些空闲的 Worker 上去执行。如图中的第二步，此时 Map 和 Reduce 的进程都已经准好，接下来就要开始处理数据了。

接下来，Worker 就要开始读取数据了，也就是读取对应的 split[x] ，并且进行处理，处理后的结果会被进行缓存。这里的读取是并发读取，也就是多个 Worker 并发的读取 split[x] 的数据。但是请注意 “Moving Computation is Cheaper than Moving Data”，这段话来自于 HDFS 的架构设计: HDFS Architecture Guide 。这句话的意思就是说，我程序运行的时候，这些所需要的数据在哪里 ? 如果和 CPU 在一台机器上的话，那么 OKay. 没有什么问题，不存在数据的传输消耗; 但是如果数据和 CPU 这样的计算资源不在一台机器上怎么办? 假设你的 CPU 在北京，而所需要的数据在上海，那么首先就需要把上海的这些数据通过网络传输到北京的机器上，然后利用北京这台机器的 CPU 进行计算，所以移动计算资源远比移动数据要节省的多，不过既然是分布式系统，这些情况就不可避免。结尾会介绍 Locality 特性。

然后，这些处理后被缓存的 pairs 会周期性的写入磁盘，并且由 partioning function 把这些处理后的 pairs 分区到 R 个 region 中。这里涉及到一个概念，就是 hash partioning function，我来解释一下，这里注意，论文中说，把这些处理后的 pairs 分区到 R 个 region 里，那么问题来了，这些数据到哪些 region? 是胡乱的乱跑吗? 肯定不是。MapReduce 编程模型采用了 hash(key) mod R 这样的分区函数对处理后的数据进行分区。那么很显然，具有相同 key 的数据会被发送到同一台机器上。所以这样的设计很棒棒呢！即使你之前相同的 key 一个在北京，另一个在上海，那么这些处理后的中间结果的 key 都会在同一台机器上，显然方便往后的操作。此时，这些数据的地理位置，或者说物理位置，或者说 IP 地址都已经发生了变化。所以这些消息需要发送给 Master 结点以汇报自己所在的地址，以方便 Master 通知 reduce worker 去到正确的地理位置取数据。所以在 Intermediate files 里面的每个结点上，这些机器里面的所有的 key 都是相同的。

数据如下所示，已经排序好：

(sup, 1)
(sup, 1) // hash("sup") mod R = 3 [假设等于 3，那么被分配到 3 号机器上，已经落地]

(duck, 1)
(duck, 1)
(duck, 1) // hash("duck") mod R = 6 [假设等于 6，那么被分配到 6 号机器上，已经落地]

(a, 1)
(a, 1) // hash("a") mod R = 12 [假设等于 12，那么被分配到 12 号机器上，已经落地]

(bunch, 1)
(bunch, 1)
(bunch, 1) // hash("bunch") mod R = 9 [假设等于 9，那么被分配到 9 号机器上，已经落地]

接下来，Master 结点告诉 reduce worker 所需要的数据在什么位置，然后 reduce worker 使用 RPC 机制去读取数据。reduce worker 对于每一个遇到的中间结果的 key 以及对应的 value 的集合发送给 reduce 函数。reduce 函数执行后的结果数据集为

(duck, 3)
(a, 2)
(bunch, 3)

到此结束。

简单扩展下这里的 hash(key) mod R

在这篇论文里，默认的分区函数采用的是 hash(key) mod R，这样产生的结果倾向于一个非常平衡的分区，从而不会导致数据发送倾斜。假设此时有另一个需求，我们需要所有来自同一个主机的 URL 的请求最后全部落入同一个机器的同一个文件中，那么就可以把该 hash 函数设计成 hash(Hostname(urlkey)) mod R, 这样来自于同一台机器的 url 请求的所有计算结果落入同一台机器中。

Locality

“Moving Computation is Cheaper than Moving Data”，所以 MapReduce 编程模型设计的时候肯定也考虑到了这个情况。在 MapReduce 模型中，Master 会试图把 Map Task 的任务调度到离它最近的数据源的位置，这里的最近，应该就是地理位置最近的位置。毕竟无法做到 CPU 和该 CPU 所要计算的数据在一台机器上，也因为是分布式系统，必定会存在数据传输，所以最大化的从最近的结点拉数据，然后进行计算。

在实际的应用中，当 MapReduce 在集群中运行的时候，大部分的数据都是从本地读取并且处理这些数据并没有什么网络的带宽。
