MapReduce 论文读后感

发表于 2017-05-20 |

如何去读一篇论文

阅读论文与阅读博客和小说不一样，以下提供了一些链接，告诉你如何去阅读一篇论文。

  • How to read an academic article
  • Advice on reading academic papers
  • How to read and understand a scientific paper
  • Should I Read Papers?
  • The Refreshingly Rewarding Realm of Research Papers


这是我接触分布式系统领域读的第一篇论文，在论文的 Abstract 里面就说了 MapReduce 是一种编程模型，并且这种编程模型用来处理并生成大量的数据集。下面均为我个人对该论文的理解所写，如果有错，还希望指出。

用户定义一个 map 函数用来处理 key/value pairs, 并且在处理结束后生成一个中间的 key/value pairs. 这很好理解，举例说明，假设我有如下的数据集。

sup sup
duck duck thanks a bunch
thanks a bunch

当把此编程模型的 map 函数用在该数据集后，会产生如下的输出结果, 可以从下面的数据集看出来，括号内的第一个单词为 key , 第二个字母为该单词出现的次数。map 并不会对这些结果最汇总，只是单纯的对输入集做一个简单记录而已。每读到一个单词，就会标记出现了一次。论文里把下面的数据集叫做 intermediate key/value pairs。接下来，MapReduce 函数的编程模型就会把该数据集发送给 reduce 函数。

(sup, 1)
(sup, 1)
(duck, 1)
(duck, 1)
(duck, 1)
(thanks, 1)
(a, 1)
(brunch, 1)
(thanks, 1)
(a, 1)
(brunch, 1)

reduce 函数在接受到上述的数据集后，就会做一件事，论文里面是这样说的：accepts an intermediate key i and a set of values for that key. It merges together these values to form a possibly smaller set of values. 按照这样的理解，对于 WordCount 这样的业务需求，我在 reduce 里面所做的事情应该就是把相同 key 的 value 进行合并，也就是说求和。做一个极端思考: 假设此时，对于 key = sup 这样的数据在 A 机器上 [中国]，而对于 key = duck 这样的数据在 B 机器上 [澳大利亚]。那么在做最后结果汇总的时候就难免会因为网络传输所带来的序列化和反序列化的操作，那么这样会造成很大的 IO 开销以及带来各种问题，当然还有很多别的问题都会产生，比如时钟的问题，见这篇论文 Time, Clocks, and the Ordering of Events in a Distributed System，海底电缆被鲨鱼咬断等等等等各种问题，这也就是分布式系统领域困难的地方。

按照上面的说法，reduce 函数在处理之后，会产生如下的数据集, 这也是我们最后需要的结果。

(sup, 2)
(duck, 3)
(thanks, 2)
(a, 2)
(brunch, 2)

同样的，MapReduce 编程模型还可以用在很多别的场合，比如我需要统计网站不同网页的访问次数。

map 函数用来处理每次用来请求的地址，并且输出 (URL, 1)， 比如下面的：

(http://www.vip.com/, 1)
(http://www.vip.com/, 1)
(http://kid.vip.com/, 1)
(http://act.vip.com/22263.html, 1)
...
...

再经过 reduce 函数输出最后的统计结果

(http://www.vip.com/, 2)
(http://kid.vip.com/, 1)
(http://act.vip.com/22263.html, 1)
...
...

开源社区里面常用的 Git 操作

Git Commands in Github & Gitlab & Git [ 这篇文章不适合 Git 新手 ]

最常用的 Git 操作命令 一般有 git add, git commit, git fetch, git rebase, git reset, git push, git branch, git log, git rm, git checkout, git remote 以及加在这些命令后面的参数，然后其它的我好像也没有怎么用过了。我不会讲解每个命令的具体用法，只说我是如何在工作中使用它们的，并且如何用这些命令来完成看似比较复杂的开源社区以及在公司里面如何同时在 10 个左右的分支上同时开发项目的，以及在子分支上建立 subtask 再一步一步 merge 到 master / dev / trunk 分支上。

git add

太简单，不想说了。

git commit

太简单，不想说了。

git fetch

每天早上打开电脑的第一件事，一般就是执行该命令。

我经常这样做：

git fetch upstream master 【社区】
git fetch 【公司】

git rebase

每天早上打开电脑的第二件事，一般就是执行这条命令；但是你可以不立马这样做，什么时候做呢？在你提交到远程分支之前进行 rebase, 如果开发的 patch 比较大，那么需要定期 rebase 远程分支代码到本地开发分支，不然最后一次性 rebase 容易导致爆炸性冲突，麻烦呢。 所以养成定期 rebase 的习惯不失为一件坏事情。

我经常这样做：

git rebase origin/dev 【公司】
git rebase upstream/master 【社区】

git reset

对于这个命令，我一般用它来 squash commit logs。 当然你也可以用 git rebase -i hashcode 去 squash logs, 不过用 rebase 的方式太过于麻烦。 所以用 git reset --soft HEAD~n [n belongs to a number], 当执行完这个命令后，不要忘记 git commit, 不然你所有的改动将会丢失哦 ~ 小心点。

我经常这样做：

git reset --soft HEAD~3
git reset --soft HEAD~8

git push

我一般用它完成本地代码推送到远程服务器上操作，还有就是新建一个分支，推送分支到远程服务器。

git log

查看提交历史，我主要用来检查 rebase 是否成功，hashcode 的值是否和远程的一致，如果一致则说明 rebase 成功，如果不一致，说明 rebase 失败。

社区操作常用的 7 个步骤，以 flink 为例

第一步：

git remote add upstream https://github.com/apache/flink

第一步如果加错了，可以执行如下的命令删除即可

git remote remove upstream

第二步：

git remote -v

第三步：

git fetch upstream master

第四步：

git rebase upstream/master

第五步：
git log 查看是否更新成功

第六步：

git push origin master

第七步：
打开自己的 github 查看是否成功
